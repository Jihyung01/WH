# /audit — 읽기 전용 구현 현황 감사

당신의 임무는 **코드베이스를 조사하고 보고서를 작성**하는 것이다. **절대 코드를 수정하지 말 것**.

## 입력
사용자가 `/audit <scope>` 형식으로 호출. `<scope>`는 감사 범위:
- 기능명 (예: "탐험 일지", "걸음수 보상")
- 파일 경로 (예: "src/services/")
- 영역명 (예: "AI 기능 전체", "결제 시스템")
- 목록 (여러 항목 쉼표 구분)

## 실행 순서

### 1단계: 범위 해석
사용자 입력을 구체적 조사 항목으로 변환하여 확인 요청:

```markdown
감사 범위 확인:

"<사용자 입력>"을 다음과 같이 해석했습니다:
1. <item-1>: <관련 파일/영역>
2. <item-2>: <관련 파일/영역>
3. <item-3>: <관련 파일/영역>

이대로 진행하면 되나요? 추가/제외할 항목이 있나요?
```

**이 단계 생략 금지**. 범위 합의 후에만 진행.

### 2단계: 조사 실행
각 항목마다 다음 순서로 조사:

1. **블루프린트 참조**: `WHEREHERE_PROJECT_BLUEPRINT.md`에서 해당 기능이 언급됐는지
2. **코드 검색**:
   ```bash
   grep -rn "<keyword>" --include="*.ts" --include="*.tsx" src/ app/
   grep -rn "<rpc-name>" supabase/
   ```
3. **실행 경로 추적**: UI 버튼 → 핸들러 → service → API → RPC/Edge Function
4. **상태 판정**:
   - ✅ **구현됨**: 전체 흐름이 연결되어 있고 정상 동작 가능
   - ⚠️ **부분 구현**: 일부 연결 빠짐 (예: UI만 있고 API 없음, 또는 반대)
   - ❌ **미구현**: 관련 코드 없음
   - 🐛 **버그 있음**: 구현됐으나 명백한 오류 존재

### 3단계: 증거 수집
각 판정은 **반드시 파일:라인 증거**를 포함해야 한다.

```markdown
증거 예시:
- `app/journal.tsx:145` — createJournal 버튼 onClick이 비어있음 (빈 함수)
- `src/services/journalService.ts` — 파일 없음
- `supabase/functions/generate-journal/index.ts:87` — 정상 구현
```

**"증거 없음"도 유효한 증거**: 관련 코드를 찾지 못했다면 솔직하게 그렇게 표기.

### 4단계: 보고서 작성
프로젝트 루트에 `AUDIT_REPORT.md` 파일 생성 (이미 존재하면 overwrite 전에 사용자 확인):

```markdown
# 감사 보고서: <scope>

> 작성일: <YYYY-MM-DD HH:mm>
> 감사자: Antigravity Agent (<model-name>)
> 범위: <사용자 입력 원문>

## 요약

| 상태 | 개수 |
|---|---|
| ✅ 구현됨 | N |
| ⚠️ 부분 구현 | N |
| ❌ 미구현 | N |
| 🐛 버그 있음 | N |

## 상세

### 1. <항목명>
**상태**: <🎯 판정>

**조사 경로**:
- UI 진입점: `<file>:<line>` — <설명>
- 핸들러: `<file>:<line>` — <설명>
- API: `<file>:<line>` — <설명>
- 서버: `<file>:<line>` — <설명>

**판정 근거**:
<증거 요약>

**예상 문제** (해당 시):
<문제 설명>

**추천 조치**:
<구체적 제안>

---

### 2. <항목명>
...

## 연관 이슈 패턴
감사 과정에서 발견한 횡단적 문제:
- <pattern-1>
- <pattern-2>

## 우선순위 제안
수정 착수 순서 제안 (영향도 × 긴급도):
1. 🔥 <highest-priority>
2. 🔥 <...>
3. ⚠️ <...>
```

### 5단계: 완료 보고
사용자에게 결과 요약 및 다음 단계 제안:

```markdown
감사 완료. `AUDIT_REPORT.md` 생성됨.

핵심 발견:
- 🐛 긴급: <urgent-1>
- 🐛 긴급: <urgent-2>
- ⚠️ 부분: <partial-1>

우선 수정이 필요한 항목에 대해 `/fix-ota <항목>` 또는 별도 
에이전트에 수정 작업 배치를 권장합니다.
```

## 절대 하지 말 것

- ❌ 코드 수정 (읽기 전용)
- ❌ 커밋 생성
- ❌ 파일 생성 (AUDIT_REPORT.md 예외)
- ❌ 추측으로 판정 ("아마 구현됐을 것" 금지)
- ❌ 증거 없는 "미구현" 판정
- ❌ 감사 범위 확인 단계 생략

## 시간 관리

- 각 항목당 조사 시간 최대 10분
- 전체 감사 60분 초과 예상 시 중간 보고 + 사용자에게 축소 여부 확인
- Grep 결과가 100건 이상이면 패턴을 좁혀서 재검색
